# Geobattery — interactive 3-D lab

Interactive 3-D viewer for a full-3D conjugate-heat-transfer study of seasonal heat storage
in the flooded **Reiche Zeche** research drift (TU Clausthal, MOVIE project). Companion to
the ALERT Geomaterials 2026 poster *"Geobattery: a dimensionless criterion for seasonal heat
storage in flooded mine workings."* The geometry is the real surveyed drift; the duty cycle
is hypothetical.

**Live viewer:** https://aliahmadi-tuc.github.io/geobattery-3d/
**Simple fallback viewer:** https://aliahmadi-tuc.github.io/geobattery-3d/simple/

Authors: **Ali Ahmadi**, Eleni Gerolymatou.

---

## How the site works (read this before restyling)

- **Single-page, no build step.** The entire app is `index.html` — inline CSS + inline JS.
  It is served raw by GitHub Pages (`.nojekyll`). There is no bundler, no framework, no npm.
- **Rendering:** [vtk.js 36.4.1](https://kitware.github.io/vtk-js/) loaded as a UMD bundle
  from unpkg (`<script src="https://unpkg.com/vtk.js@36.4.1/vtk.js">`, global `vtk`).
  The 3-D canvas is created by `vtkFullScreenRenderWindow` inside `<div id="view">`.
- **Data** is fetched at runtime from `data/<scenario>/…` (see the data contract below).
  Scenario = `drift` (default) or `array`, selected via the `?scene=` URL query — the
  scenario dropdown reloads the page with that query.
- **Deploys** are just pushes to `main`; GitHub Pages rebuilds in ~30 s.

### File tree

```
index.html          the whole app (CSS + HTML panel + JS)
data/
  drift/            as-built drift scenario
    rock.vtp        rock surface  (point arrays: T [°C], vm [MPa von Mises])
    concrete.vtp    concrete plugs (T, vm)
    air.vtp         air-filled end galleries (T)
    water_transient.vtp  water surface, one T-array per time frame (T00…T03)
    frames.json     transient manifest: [{file, label}] per time frame
    frames/T_0*.vti temperature volume per frame (Float32+zlib, sentinel −100 = outside mesh)
    flow.json       {clip, umax, Q_out, t_res_d, inlet[3], outlet[3]}
    flow_arrows.vtp pre-glyphed static velocity arrows (point array: speed [m/s])
    flowlines.json  {ds, clip, n, lines:[{pts[[x,y,z]…], spd[…]}]} inlet→outlet streamlines
    stress.json     {vmin, vmax} MPa for the von Mises colourbar
    temperature.vti (legacy, unused by the viewer)
  array/            10 m array-cell scenario — same contract, but has NO air.vtp,
                    NO flow*, NO stress.json (viewer auto-hides those features)
simple/index.html   minimal <model-viewer> fallback (models/*.gltf)
models/*.gltf       two baked models for the simple viewer
lab/index.html      redirect stub to the root (legacy URL)
```

All `.vtp`/`.vti` must stay **Float32 + zlib-compressed** — vtk.js cannot read Float64
uncompressed appended data. Data files are regenerated by Python exporters that live in a
separate (private) simulation repo; do not hand-edit them.

### index.html structure

| Section | What it does |
|---|---|
| `<style>` | full theme. CSS custom props in `:root` (`--bg --panel --ink --mut --grn --line`). Dark floating panel + colourbar chip over a light 3-D scene. |
| `#panelBtn` | ☰ button toggling the options panel (`#panel.closed`); panel starts closed under 760 px. |
| `#panel` | the options popup. Each `.grp` section folds by clicking its `h2` (class `fold`). |
| `.cbar` | dynamic colourbar (`#cbGrad #cbLo #cbMid #cbHi`), driven by `updateColorbar()`. |
| `#status` | status/error chip (bottom left). |
| `<script>` | everything: scenario resolution, layer loading, volume + transient frames, clip & slice plane, colour modes, flow arrows + moving-arrow particle system, colourbar. |

Key JS entry points (all inside `main()`):
`loadLayer` (surface layers + per-material Phong), `loadVolume`/`showFrame` (transient
volume frames), `applyCut` (one plane: clips surfaces + positions the temperature-cap
slice), `setColorMode` (material / temperature / vm / flow — also toggles which
sub-controls and actors are visible via `setFieldVis`/`syncFlowVis`), `loadFlow` /
`loadParticles`/`stepParticles` (static + moving arrows), `updateColorbar`.

### Invariants — a redesign must not break these

1. **Element ids** are the API between HTML and JS. Keep every `id=` (`view, panelBtn,
   panel, scene, sceneLabel, colMode, tempCtl, flowCtl, volOn, volInt, timeGrp, timePlay,
   timeLabel, timeSlider, layers, clipOn, sliceOn, cutAxis, clipFlip, cutPos, flowOn,
   flowMove, repr, edges, reset, cbGrad, cbLo, cbMid, cbHi, status`). Restyle freely,
   rename/move nothing.
2. `#view` must remain a full-viewport container for the WebGL canvas.
3. The `data/…` fetch paths and JSON shapes are a contract with the exporters.
4. Layer checkboxes/sliders inside `#layers` are **generated** by `buildRow()` in JS —
   style them via CSS selectors, don't hardcode rows in HTML.
5. The `flow` / `vm` options of `#colMode` are removed at runtime when a scenario lacks
   that data — don't assume they exist.
6. Keep the site framework-free and single-file; external assets only from CDNs that
   allow hotlinking (currently only vtk.js from unpkg).

### Free to change (designer's playground)

Typography, colours, spacing, the panel look (cards/tabs/accordion), the colourbar chrome,
button/slider styling, the header, the scenario switcher UI, mobile layout, icons,
loading states, the status chip — anything visual. The current design is engineer-made
and deliberately minimal: dark translucent panel + light scene.

### Planned functional work (context for design decisions)

- Velocity mode will dim everything except a barely-visible water shell and fly the
  camera to a preset view.
- The transient will get ~4 sub-steps per phase with per-phase colour scaling.
- An intro "construction" animation (mesh builds up → warm water flows → zoom out →
  hand over control) is planned as the landing experience.
- A walk-through (fly-inside) camera mode is planned.
